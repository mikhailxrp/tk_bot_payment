# Phase 2 — Базовый бот: /start и меню подписки

## Статус

✅ Готово | Начата: 2026-07-04 | Завершена: 2026-07-04 | Ветка: `phase-2` | PR: —

## Цель

Поднять рабочий grammY-бот: `/start` регистрирует/обновляет пользователя в БД и показывает
меню оформления подписки с суммой из настроек; появляется middleware `isAdmin` и заглушка
команды `/admin`, доступная только администраторам. Реальная оплата (Робокасса), выдача
invite-ссылок, mute и напоминания — в следующих фазах.

## Контекст

Опирается на Фазу 1 (✅): Prisma-схема (`User`, `Admin`, `Setting`), сид (5 ключей настроек,
первый `Admin` из `ADMIN_ID`), `config.ts` (zod, BigInt для `GROUP_ID`/`ADMIN_ID`), pino-логгер.
Разделы PRD: 3.1 (оформление подписки, happy path — без шагов 4–9, они начинаются с Фазы 3),
4 (доступ `/admin` по `telegram_id` в таблице `admins`).

**Решение, принятое на этапе проверки документации:**

- **#1 — `isAdmin` проверяет БД, не `config.ADMIN_ID`.** Middleware ищет `ctx.from.id` в таблице
  `Admin` (BigInt-сравнение) при каждом вызове, а не сравнивает с `config.ADMIN_ID` напрямую.
  Иначе после того как в Фазе 8 через панель добавят второго админа, бот его не узнает без
  хардкода. Следует из правила CLAUDE.md п.12 (данные читать из БД, не кэшировать) и PRD п.4.

### Task 2.3 — Middleware isAdmin + skeleton /admin

- **Статус:** ✅ Готово
- **Workspace:** bot
- **Цель:** middleware проверяет `ctx.from.id` против таблицы `Admin` (BigInt-сравнение, запрос
  к БД на каждый вызов); `/admin` доступен только админам, отвечает заглушкой (без ручной
  проверки/сводки/ссылки на панель — это появится в Фазах 5–8).
- **Файлы:** `apps/bot/src/bot/bot.ts` (регистрация middleware), `apps/bot/src/bot/handlers/admin.ts`
- **Out of scope:** ручная проверка подписок (Фаза 5), сводка платежей (Фаза 6), ссылка на
  панель (осмысленна только после Фазы 7)
- **DoD:**
  - [x] админ (есть в таблице `Admin`) получает ответ на `/admin`
  - [x] не-админ получает отказ без падения бота
  - [x] проверка идёт запросом к БД, а не сравнением с `config.ADMIN_ID`
