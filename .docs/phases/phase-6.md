# Phase 6 — Ежедневный cron: mute истёкших (только закрытая группа)

## Статус

🔄 В работе | Начата: 2026-07-06 | Ветка: `phase-6`

## Цель

Раз в сутки (и по кнопке в `/admin`) бот находит пользователей закрытой группы с истёкшей
подпиской (`status=ACTIVE`, `expiresAt < now`), ограничивает им отправку сообщений
(`restrictChatMember`, не `banChatMember`), переводит в `MUTED`, шлёт им личное сообщение с
кнопкой оплаты и сводку админам. При следующей успешной оплате такого пользователя мьют
снимается автоматически (расширение транзакции webhook из Фазы 3). Общая группа
(`CommonAccess`) этим никогда не затрагивается. `/admin` перестаёт быть заглушкой и получает
все три функции из PRD §4: ручную проверку, быструю сводку и ссылку на панель. Напоминания
(PRD 3.4) — вне скоупа, это Фаза 7.

## Контекст

Опирается на Фазу 5 (✅): паттерн best-effort Telegram-вызовов после коммита транзакции
(лог + алерт админам при сбое, как `grantAccessBestEffort`), `notifyAdmins` с троттлингом,
`formatUserMention`. Разделы PRD: 3.3 (mute истёкших), 3.4 (только структура dailyCheck —
напоминания сами реализуются в Фазе 7), 3.5 (unmute при оплате), 4 (функции `/admin`), 8
(нефункциональные требования). Правила CLAUDE.md пп. 6 (mute, не ban), 9 (продление/сброс
флагов — не меняется), 11 (рассылки через `notify.ts`), 12 (настройки не кэшировать).

**Prisma-схема готова с Фазы 1** (`UserStatus.MUTED`, `User.mutedAt`) — миграция в этой фазе
не требуется.

**Решения, принятые на этапе проверки документации (обсуждены и подтверждены перед стартом):**

- **#1 — `/admin` в этой фазе получает все три функции PRD §4**, не только кнопку ручной
  проверки: 🔄 «Проверить подписки», 📊 «Сводка» (активные/замьюченные/оплаты за сегодня),
  🔗 «Панель» (url-кнопка на `ADMIN_PANEL_URL`). Причина: `phases.md` называл в Фазе 6 только
  ручную проверку, а сводка и ссылка на панель не были закреплены ни за одной фазой во всей
  карте; обе фичи не зависят от ещё не построенной Next.js-панели (Фазы 8–10), а `/admin`
  и так переписывается со stub на реальный функционал именно здесь.
- **#2 — Unmute читает `bot.api.getChat(GROUP_ID).permissions` и передаёт объект целиком** в
  `restrictChatMember`, а не хардкодит права. Причина: непереданные поля `ChatPermissions`
  Telegram трактует как `false` — хардкод (например, только `can_send_messages: true`) молча
  урезал бы права относительно реальных настроек группы.
- **#3 — Планировщик тикает каждую минуту** (`node-cron`, `* * * * *`, `timezone:
  'Europe/Moscow'`), на каждом тике читает `Setting.cron_time` из БД (без кэша) и сравнивает с
  текущим отформатированным `HH:mm`; guard на «не чаще раза в календарный день» — в памяти
  хранится дата последнего запуска. Причина: `node-cron` регистрирует cron-выражение один раз
  при `schedule()` и не имеет автоматического reschedule по изменению в БД — фиксированное
  выражение потребовало бы рестарта бота при смене `cron_time` из панели (Фаза 9), что
  противоречит правилу «настройки не кэшировать».
- **#4 — `runDailyCheck()` защищена MySQL `GET_LOCK()`/`RELEASE_LOCK()`** (неблокирующий,
  таймаут `0`), не in-process флагом. Acquire/работа/release оборачиваются в один
  `prisma.$transaction`, чтобы лок и снятие лока гарантированно шли на одном соединении (лок
  MySQL — на уровне сессии/соединения; отдельные вызовы через пул соединений Prisma рискуют
  взять и снять лок с разных сессий, и защита не сработает). Конкурентный вызов (тик cron во
  время ручного запуска из `/admin` или наоборот) получает отказ сразу и пропускается с
  логом. Решение пересмотрено при планировании Task 6.1: `dod-global.md` явно требует именно
  `GET_LOCK`/`RELEASE_LOCK` (не in-process) для защиты cron/ручного запуска — переживает
  рестарт процесса, в отличие от булева флага в памяти (изначальное решение прошлой сессии).

## Таски

### Task 6.1 — `jobs/dailyCheck.ts`: mute истёкших + сообщение с оплатой + сводка админам

- **Статус:** ✅ Готово
- **Workspace:** bot
- **Цель:** общая функция `runDailyCheck()`, одинаково вызываемая cron и ручной кнопкой,
  защищённая изнутри неблокирующим MySQL `GET_LOCK()`/`RELEASE_LOCK()` (решение #4) —
  конкурентный вызов сразу получает отказ, а не встаёт в очередь; выборка `User` где
  `status=ACTIVE AND expiresAt < now`; каждому — `restrictChatMember(GROUP_ID,
  userId, { can_send_messages: false })`, `status=MUTED`, `mutedAt=now`; личное сообщение
  «Подписка истекла, оплатите продление» с кнопкой «Оплатить» (новый `Payment(PENDING,
  product=SUBSCRIPTION)` + `buildPaymentUrl`, по аналогии с `handleSubscribeCallback`); по
  завершении — сводка админам через `notifyAdmins` (сколько замьючено + список через
  `formatUserMention`), отправляется всегда, включая случай «замьючено 0».
- **Файлы:** `apps/bot/src/jobs/dailyCheck.ts` (новый), `apps/bot/src/services/subscription.ts`
  (вынести общую функцию создания `Payment` + ссылки на подписку, переиспользуемую `start.ts`
  и `dailyCheck.ts`; функция мьюта одного пользователя), `apps/bot/test/dailyCheck.test.ts`
  (новый)
- **Out of scope:** напоминания активным/замьюченным (Фаза 7), unmute при оплате (Task 6.2),
  cron-обвязка и `/admin` (Task 6.3)
- **DoD:**
  - [x] конкурентный вызов `runDailyCheck()` (например, из теста — два параллельных вызова)
        получает `GET_LOCK(..., 0)` = `0` на втором вызове и завершается без обработки строк,
        с логом; `RELEASE_LOCK` вызывается в `finally` и подтверждается тестом
  - [x] `ACTIVE` + `expiresAt < now` → `restrictChatMember(can_send_messages:false)`,
        `status=MUTED`, `mutedAt` проставлен
  - [x] `ACTIVE` + `expiresAt` в будущем — не трогается (граница: `expiresAt === now` —
        мьютится [реализовано через `lte`, см. «Итоги» в `TASK.md`], `expiresAt = now + 1с` — нет)
  - [x] пользователь только с `CommonAccess` (без активной `SUBSCRIPTION`) не затрагивается
        никогда
  - [x] уже `MUTED`/`NEW`/`LEFT` — повторно не обрабатывается (идемпотентность повторного
        запуска в тот же день)
  - [x] каждому замьюченному — личное сообщение с кнопкой «Оплатить» на реальную
        Робокасса-ссылку (новый `Payment(PENDING)`)
  - [x] сбой отдельного Telegram-вызова (`restrictChatMember`/`sendMessage`) на одном юзере
        логируется и не прерывает обработку остальных
  - [x] по итогам — `notifyAdmins` с текстом «замьючено N» + упоминания (`formatUserMention`),
        включая `N=0`
