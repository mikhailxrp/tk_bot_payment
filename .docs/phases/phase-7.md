# Phase 7 — Напоминания (оба типа, только закрытая группа)

## Статус

✅ Готово (7.1–7.3 ✅, 7.4 авто ✅, ручная проверка ⏳) | Начата: 2026-07-06 | Ветка: `phase-7`

## Цель

Тот же ежедневный прогон (`runDailyCheck`, cron + кнопка `/admin`) после mute истёкших рассылает
два типа напоминаний по закрытой группе: **активным** — за `remind_days`=3 дня до `expiresAt`,
один раз за период (`reminderSentAt`); **замьюченным** — каждые `muted_remind_days`=10 дней после
mute, до оплаты (`lastMutedRemindAt`). Оба напоминания несут персональную ссылку оплаты с кнопкой.
Сброс обоих флагов при оплате уже реализован (`applyPayment`, Фаза 3/6) — эта фаза платёж не
трогает. Общая группа (`CommonAccess`) напоминаниями не затрагивается никогда. По итогам прогона
админам уходит сводка с числом замьюченных и числом отправленных напоминаний каждого типа.

## Контекст

Опирается на Фазу 6 (✅): `runDailyCheck()` под MySQL `GET_LOCK`/`RELEASE_LOCK` в одной
`prisma.$transaction`, паттерн «DB-guard в транзакции → Telegram best-effort после коммита»
([dailyCheck.ts](../../apps/bot/src/jobs/dailyCheck.ts)), `createSubscriptionPaymentLink`
(персональный `Payment(PENDING)` + `buildPaymentUrl`), `paymentKeyboard`, `formatUserMention`,
`notifyAdmins`. Флаги `reminderSentAt`/`lastMutedRemindAt`/`mutedAt` есть с Фазы 1 — **миграция не
нужна**. `applyPayment` уже сбрасывает `reminderSentAt` и `lastMutedRemindAt` в `null` при оплате
([subscription.ts:71-79](../../apps/bot/src/services/subscription.ts#L71-L79)).

Разделы PRD: 3.4 (таблица напоминаний), 3.2/3.5 (сброс флагов при оплате — уже есть), 8
(рассылки ≤25 msg/sec, 403 не роняет джобу). Правила CLAUDE.md пп. 9 (сброс флагов), 10 (оба типа
напоминаний), 11 (рассылки только через `notify.ts` с троттлингом), 12 (настройки не кэшировать).

**Решения, принятые на этапе проверки документации (подтверждены перед стартом):**

- **#A — `notify.ts` получает троттлированный per-recipient sender.** Текущий
  `sendThrottledTextMessages` шлёт один и тот же текст без `reply_markup` и не подходит для
  персональных напоминаний с кнопкой оплаты; цикл part-1 (`sendExpiredSubscriptionMessage`) шлёт
  сообщения вообще без троттлинга, в обход `notify.ts` — расхождение с CLAUDE.md п.11. Вводится
  `sendThrottledPersonalMessages(bot, items: { chatId, text, reply_markup? }[], options?)`:
  ≤25 msg/sec, ошибка одного получателя (403 и др.) логируется и не прерывает батч. На него
  переводятся И напоминания обоих типов, И существующая рассылка mute-сообщений part-1.
- **#B — Точка отсчёта первого напоминания замьюченным — `mutedAt`.** Условие срабатывания:
  `reference = lastMutedRemindAt ?? mutedAt`; напоминание уходит, если `now >= reference +
  muted_remind_days*день`; при отправке `lastMutedRemindAt = now`. Даёт матрицу DoD (mute 9 дней —
  нет, 10 — да, 20 — снова да).
- **#C — Часть 2 (напоминания) считается после части 1 (mute), в той же транзакции под тем же
  `GET_LOCK`.** Свежезамьюченный в этом же прогоне имеет `mutedAt = now` (разница 0 < 10 дней) —
  напоминание замьюченному ему не уйдёт; активные напоминания селектят `status=ACTIVE`, а
  свежезамьюченные уже `MUTED` — тоже не попадут. Так один прогон не шлёт пользователю сразу и
  «подписка истекла», и напоминание.
- **#D — Флаги ставятся в транзакции ДО отправки (best-effort, без ретраев).** По паттерну Фазы 6:
  `reminderSentAt`/`lastMutedRemindAt` проставляются внутри `$transaction`, отправка идёт после
  коммита. При сбое отправки: активный не получит напоминание за этот период; замьюченный получит
  в следующий 10-дневный цикл. Компромисс принят осознанно (идемпотентность важнее гарантии
  доставки, как для invite-ссылок и mute-сообщений).
- **#E — Граница «за 3 дня» включительна.** Активное напоминание: `status=ACTIVE AND expiresAt >
  now AND expiresAt <= now + remind_days*день AND reminderSentAt IS NULL` (`lte`, как mute).
- **#F — Сводка админам расширяется счётчиками напоминаний.** К существующей строке «замьючено N»
  добавляется «напоминаний отправлено: активным X, замьюченным Y». Счётчики — по числу получателей,
  которым напоминание фактически отправлено (не по числу отобранных под флаг).
- **#G — Невалидный/отсутствующий `remind_days`/`muted_remind_days` (проверено при старте Task
  7.2).** Соответствующий тип напоминаний пропускается в этом прогоне (warning-лог), mute part-1 и
  другой тип напоминаний не затрагиваются — прогон не падает и не подставляет дефолт из PRD
  молча (решение подтверждено перед реализацией 7.2).

**Настройки читаются живьём из `Setting` на каждом прогоне** (`remind_days`, `muted_remind_days`) —
не кэшировать (CLAUDE.md п.12). Оба напоминания используют `subscriptionBot` и закрытую группу; ни
`commonBot`, ни `CommonAccess` не затрагиваются. Пользователь с одновременно активной подпиской
(`status=ACTIVE`) и `CommonAccess` получает напоминания по подписке штатно — селект по `status`
уже корректно его включает.

### Task 7.2 — Отбор кандидатов напоминаний + проставление флагов в транзакции

- **Статус:** ✅ Готово (2026-07-07)
- **Workspace:** bot
- **Цель:** в `services/subscription.ts` — чистые/транзакционные функции отбора и пометки
  кандидатов, читающие `remind_days`/`muted_remind_days` из `Setting` живьём. Активные:
  `selectAndMarkActiveReminders(tx, now, remindDays)` — `status=ACTIVE AND expiresAt > now AND
  expiresAt <= now + remindDays*день AND reminderSentAt IS NULL`, для отобранных
  `reminderSentAt = now` (решения #E, #D). Замьюченные:
  `selectAndMarkMutedReminders(tx, now, mutedRemindDays)` — `status=MUTED AND (lastMutedRemindAt IS
  NULL ? mutedAt : lastMutedRemindAt) <= now - mutedRemindDays*день`, для отобранных
  `lastMutedRemindAt = now` (решения #B, #D). Обе функции вызываются в той же транзакции `runDailyCheck`
  **после** mute, под тем же `GET_LOCK` (решение #C), и возвращают списки `{ id, username }` для
  последующей отправки. Настройки читаются в `runDailyCheck` перед транзакцией (без кэша).
- **Файлы:** `apps/bot/src/services/subscription.ts`, `apps/bot/src/jobs/dailyCheck.ts`,
  `apps/bot/test/subscription.test.ts`, `apps/bot/test/dailyCheck.test.ts`
- **Out of scope:** сам механизм отправки (Task 7.1/7.3), сводка (Task 7.3)
- **DoD:**
  - [x] активные: `expiresAt = now + 4д` — не отобран; `= now + 3д` — отобран; повторный прогон
        (уже `reminderSentAt != null`) — не отобран; `expiresAt < now` — не отобран (уже за
        границей активного окна, обрабатывается mute)
  - [x] замьюченные: `mutedAt = now - 9д`, `lastMutedRemindAt=null` — не отобран; `= now - 10д` —
        отобран; после отправки (`lastMutedRemindAt = now-10д` при `now`... т.е. прошло 10д) —
        снова отобран; свежезамьюченный этого же прогона (`mutedAt = now`) — не отобран
  - [x] пользователь только с `CommonAccess` (`status != ACTIVE/MUTED`) не отбирается ни в один
        из списков ни при каком прогоне
  - [x] флаги (`reminderSentAt`/`lastMutedRemindAt`) проставляются внутри транзакции до отправки;
        отбор и пометка идут после mute и под тем же `GET_LOCK` (конкурентный прогон пропускается)
  - [x] `remind_days`/`muted_remind_days` читаются из `Setting` на каждом прогоне; смена значения
        в БД подхватывается без рестарта
