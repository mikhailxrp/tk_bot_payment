# План фаз — Telegram Subscription Bot + Admin Panel

Правило: фаза = ветка `phase-N` → PR в `dev`. Каждая фаза имеет чёткий Definition of Done (DoD).
Зависимости строго сверху вниз, кроме отмеченных параллельных.

## Карта зависимостей

```
1 → 2 → 3 → 4 → 5 → 6 ──────────────┐
            │                        ├→ 10
            └→ 7 → 8 → 9 ───────────┘
                 (панель можно параллелить с 5–6)
```

---

## Фаза 1 — Каркас монорепо и инфраструктура

**Что:** npm workspaces (`apps/bot`, `apps/admin`, `packages/db`), подключение к реальной MySQL 8
на хостинге (без Docker/локальной установки), Prisma-схема целиком (все модели из PRD п.6),
сид (настройки + первый админ из `ADMIN_ID`), `config.ts` с zod, pino, ESLint/Prettier/tsconfig.

**Почему первой:** схема БД — общий контракт бота и панели; меняем её потом только миграциями.

**DoD:** `DATABASE_URL` на реальную БД хостинга подключается → `prisma migrate dev` → сид
отрабатывает, `npm run dev -w apps/bot` запускает пустой процесс без ошибок, невалидный `.env`
роняет старт с понятной ошибкой.

## Фаза 2 — Базовый бот: /start и меню подписки

**Что:** grammY, регистрация/обновление User при `/start` (upsert), кнопка «Оформить подписку» →
сообщение с суммой (из `Setting.price`) и кнопкой «Оплатить» (пока заглушка-URL).
Middleware `isAdmin` + скелет `/admin` (только для админов).

**DoD:** /start работает, юзер появляется в БД, цена читается из БД (меняешь в Prisma Studio —
меняется в боте без рестарта), не-админ на `/admin` получает отказ.

## Фаза 3 — Робокасса: платёжная ссылка + webhook

**Что:** `payments/robokassa.ts` — чистые функции: buildPaymentUrl (с Receipt, двойное кодирование),
verifyResultSignature. Fastify: `POST /robokassa/result` → проверка подписи → транзакция
(Payment.PAID + продление expiresAt по правилу из PRD 3.2) → ответ `OK{InvId}`. Идемпотентность.
Success/Fail URL → deep-link в бота.

**Почему отдельной фазой:** самая рискованная интеграция; чистые функции подписи покрываем
unit-тестами сразу здесь, не откладывая до фазы 10.

**DoD:** в тестовом режиме (`IsTest=1`) реальная оплата проходит, Payment → PAID, expiresAt
проставлен; повторный POST webhook не продлевает второй раз; кривая подпись → 400.

## Фаза 4 — Выдача доступа в группу

**Что:** `services/subscription.ts`: createChatInviteLink (member_limit: 1) после оплаты,
отправка ссылки юзеру; обработчик `chat_member` — фиксация вступления (`inGroup=true`,
статус ACTIVE); уведомления админам (оплата, вступление) через `services/notify.ts` (троттлинг, 403).

**DoD:** полный happy path: /start → оплата (тест) → ссылка → вступил → админам пришли
2 уведомления; ссылка второй раз не работает.

## Фаза 5 — Ежедневный cron: mute истёкших

**Что:** `jobs/dailyCheck.ts` (часть 1): выборка ACTIVE с `expiresAt < now` → restrictChatMember
(can_send_messages: false) → статус MUTED, mutedAt → личка юзеру с кнопкой оплаты → сводка админам.
Unmute при оплате (расширение транзакции фазы 3). Запуск: node-cron (Europe/Moscow) + кнопка
«Проверить подписки» в `/admin` (та же функция).

**Как проверить:** руками выставить expiresAt в прошлое → ручной запуск → юзер в группе, писать
не может; оплата → пишет снова.

**DoD:** сценарий выше + cron срабатывает по расписанию (проверка на коротком интервале в dev).

## Фаза 6 — Напоминания (оба типа)

**Что:** dailyCheck часть 2: активным за 3 дня (reminderSentAt, 1 раз за период);
замьюченным каждые `muted_remind_days`=10 дней (lastMutedRemindAt, до оплаты).
Сброс обоих флагов при оплате.

**DoD:** матрица дат проверена руками (за 4 дня — нет, за 3 — да, повторно — нет;
mute 9 дней — нет, 10 — да, 20 — снова да).

## Фаза 7 — Панель: каркас + auth

**Что:** Next.js App Router, Auth.js v5 credentials (email+bcrypt из `Admin`), layout, guard
всех роутов, страница логина. Скрипт создания пароля первому админу.

**DoD:** без сессии любой роут → /login; логин работает; чужой email — отказ.

## Фаза 8 — Панель: пользователи, платежи, настройки, админы

**Что:** таблица пользователей (фильтры по статусу, вкладка «Ограниченные»), карточка юзера с
историей платежей, действия mute/unmute/продлить вручную — через internal API бота
(`POST /internal/mute|unmute`, Bearer INTERNAL_API_TOKEN — добавляется в этой фазе в apps/bot);
список платежей; настройки (price, remind_days, muted_remind_days, cron_time); CRUD админов.
Server actions + zod + проверка сессии.

**DoD:** mute из панели → юзер реально замьючен в группе ≤ 1 мин; смена цены в панели →
бот показывает новую без рестарта; BigInt не ломает сериализацию.

## Фаза 9 — Панель: дашборд и аналитика

**Что:** выручка день/месяц, MRR, счётчики статусов, новые подписки, конверсия /start→оплата,
график платежей (recharts). SQL-агрегации через Prisma (groupBy/raw).

**DoD:** цифры дашборда сходятся с ручным SELECT по тем же периодам.

## Фаза 10 — Тесты, деплой, приёмка

**Что:** vitest: подписи (включая Receipt), расчёт продления, выборки cron (граничные даты),
webhook-интеграция (валид/невалид/повтор), internal API auth. Деплой: VPS, Nginx (2 upstream:
webhook бота + панель), PM2 (2 процесса), боевые ключи Робокассы, прогон всех критериев приёмки
из PRD п.10.

**DoD:** все критерии приёмки PRD зелёные на проде с тестовым платежом, затем 1 боевой платёж.
