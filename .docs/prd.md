# PRD — Telegram-бот платной подписки на группу

## 1. Обзор

Система из двух приложений:

1. **Telegram-бот** — продажа доступа в две группы через Робокассу:
   - **Закрытая группа** — подписка на 30 дней; ежедневная проверка истёкших подписок
     (mute, не kick), напоминания;
   - **Общая группа** — разовый платёж меньшей суммы, бессрочный доступ (без mute,
     без напоминаний).
2. **Веб-админ-панель** (Next.js) — управление пользователями, настройками,
   администраторами; статистика и аналитика.

Бот заменяет два существующих бота (подписочный и разовый), работающих на одном
магазине Робокассы, — после запуска Result URL магазина переключается на этот бот,
старые боты выводятся из эксплуатации.

**Стек:** Node.js (TypeScript), grammY, Fastify (webhook Робокассы), Next.js 15 (админка),
Auth.js v5, реальная MySQL 8 на хостинге (без Docker/локальной установки), Prisma ORM
(общий пакет), node-cron. Монорепо (npm workspaces).

## 2. Роли

| Роль          | Описание                                                           |
| ------------- | ------------------------------------------------------------------ |
| Пользователь  | Оформляет и продлевает подписку через бота                         |
| Администратор | Веб-панель: полное управление. Бот: уведомления + быстрые действия |
| Бот           | Админ обеих групп: закрытой (restrict members + invite users) и общей (invite users) |

## 3. Пользовательские сценарии (бот)

### 3.1 Оформление подписки на закрытую группу (happy path)

1. Пользователь отправляет `/start`.
2. Бот показывает приветствие + выбор группы: две inline-кнопки —
   **«Закрытая группа (подписка)»** и **«Общая группа (разовый доступ)»**
   (флоу общей группы — п. 3.6).
3. По нажатию «Закрытая группа» бот показывает сумму подписки (из настроек БД) и кнопку **«Оплатить»**.
4. Кнопка ведёт на платёжную страницу Робокассы (URL с подписью SignatureValue и чеком Receipt).
5. Пользователь оплачивает. Робокасса шлёт POST на Result URL бота.
6. Бот проверяет подпись, помечает платёж `PAID`, продлевает/создаёт подписку (+30 дней).
7. Бот создаёт **одноразовую** invite-ссылку (`member_limit: 1`) и отправляет её пользователю.
8. Если пользователь ранее был замьючен за неуплату — бот снимает ограничения.
9. Админы получают уведомление в личку: «Оплата от @user, сумма, срок до …».

### 3.2 Продление

- Тот же флоу через `/start` или кнопку из напоминания.
- Подписка активна → новый срок = текущая дата окончания + 30 дней.
- Истекла → новый срок = дата оплаты + 30 дней.

### 3.3 Истечение подписки (ежедневный cron)

1. Раз в сутки (по умолчанию 09:00 МСК) бот выбирает всех с `expires_at < NOW()` и статусом `ACTIVE`.
2. Каждому применяет `restrictChatMember`: `can_send_messages: false`
   (пользователь остаётся в группе, читает, но не пишет).
3. Статус → `MUTED`.
4. Пользователю в личку: «Подписка истекла, оплатите продление» + кнопка оплаты.
5. Админам — сводка: сколько замьючено, список.

### 3.4 Напоминания

| Кому        | Когда                     | Сколько раз                           |
| ----------- | ------------------------- | ------------------------------------- |
| Активным    | За 3 дня до `expires_at`  | 1 раз за период (`reminderSentAt`)    |
| Замьюченным | Каждые 10 дней после mute | Пока не оплатит (`lastMutedRemindAt`) |

Оба типа отправляет тот же ежедневный cron. Флаги сбрасываются при оплате.

### 3.5 Возврат после оплаты

- `MUTED` и оплатил → снятие mute (restrictChatMember с правами по умолчанию), статус → `ACTIVE`.
- Сам вышел из группы → новая одноразовая ссылка.

### 3.6 Разовый доступ в общую группу

1. `/start` → кнопка **«Общая группа (разовый доступ)»**.
2. Бот показывает сумму (из `Setting.price_common`) и кнопку **«Оплатить»** —
   платёж Робокассы с `Payment.product = LIFETIME`.
3. После webhook Result URL: в одной транзакции `Payment.PAID` + запись `CommonAccess`
   (`paidAt`). `User.expiresAt`, статусы и флаги напоминаний **не затрагиваются**.
4. Бот создаёт **одноразовую** invite-ссылку в общую группу (`member_limit: 1`) и отправляет её.
5. Cron, mute и напоминания к общей группе **не применяются** — доступ бессрочный.
6. Если доступ уже оплачен: бот сообщает об этом; если пользователь сам вышел из
   группы — выдаёт новую одноразовую ссылку (без повторной оплаты).

## 4. Админ-функции в боте (минимум)

Доступ: `telegram_id` в таблице `admins`. Команда `/admin`:

- 🔄 Запустить проверку подписок вручную (та же логика, что cron)
- 📊 Быстрая сводка: активных / замьюченных / оплат за сегодня
- 🔗 Ссылка на веб-панель

Уведомления в личку каждому админу: успешная оплата, вступление в группу, итог ежедневной проверки.

## 5. Веб-админ-панель (Next.js)

Авторизация: Auth.js v5, credentials (email+пароль) с ролью, привязка к записи `Admin`.

### 5.1 Разделы

| Раздел             | Функции                                                                                                                                                                                    |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Дашборд**        | Выручка (день/месяц), MRR, активные/замьюченные/ушедшие, новые подписки, конверсия /start → оплата, график платежей                                                                        |
| **Пользователи**   | Таблица: имя, @username, статус, срок подписки; фильтры по статусу; **отдельный фильтр/вкладка «Ограниченные (MUTED)»**; действия: mute/unmute вручную, продлить вручную, история платежей |
| **Платежи**        | Список всех платежей: статус, сумма, продукт (подписка / разовый), дата, пользователь                                                                                                      |
| **Настройки**      | Сумма подписки, цена разового доступа в общую группу, время cron, дни напоминания (3), интервал напоминаний замьюченным (10), текст сообщений                                              |
| **Администраторы** | Добавить/удалить админа (telegram_id + email для панели)                                                                                                                                   |

### 5.2 Действия панели, влияющие на Telegram

Mute/unmute из панели должны вызывать Telegram API. Реализация: панель пишет задачу в таблицу
`bot_tasks` (или дергает внутренний HTTP-эндпоинт бота, защищённый токеном) — бот исполняет и
помечает выполненной. Выбор способа — на этапе реализации; в PRD фиксируем требование: действие
из панели применяется в группе ≤ 1 минуты.

## 6. Модель данных (Prisma / MySQL)

```prisma
model User {
  id                 BigInt     @id            // telegram_id
  username           String?
  firstName          String?
  status             UserStatus @default(NEW)  // NEW | ACTIVE | MUTED | LEFT
  expiresAt          DateTime?
  reminderSentAt     DateTime?  // напоминание за 3 дня (текущий период)
  mutedAt            DateTime?
  lastMutedRemindAt  DateTime?  // цикл напоминаний каждые 10 дней
  inGroup            Boolean    @default(false)
  createdAt          DateTime   @default(now())
  payments           Payment[]
  commonAccess       CommonAccess?
}

model Payment {
  id        Int           @id @default(autoincrement()) // = InvId Робокассы
  userId    BigInt
  amount    Decimal       @db.Decimal(10, 2)
  product   ProductType   @default(SUBSCRIPTION) // SUBSCRIPTION | LIFETIME
  status    PaymentStatus @default(PENDING) // PENDING | PAID | FAILED
  createdAt DateTime      @default(now())
  paidAt    DateTime?
  user      User          @relation(fields: [userId], references: [id])
}

// Бессрочный доступ в общую группу (разовый платёж)
model CommonAccess {
  userId    BigInt   @id            // telegram_id
  paidAt    DateTime                // дата разовой оплаты
  inGroup   Boolean  @default(false)
  createdAt DateTime @default(now())
  user      User     @relation(fields: [userId], references: [id])
}

model Admin {
  telegramId   BigInt   @id
  username     String?
  email        String?  @unique  // вход в веб-панель
  passwordHash String?
  addedBy      BigInt?
  createdAt    DateTime @default(now())
}

model Setting {
  key   String @id  // price, price_common, period_days, cron_time, remind_days, muted_remind_days
  value String
}
```

Дефолты: `price=990`, `price_common=500` (уточнить, см. п.11), `period_days=30`,
`remind_days=3`, `muted_remind_days=10`, `cron_time=09:00`.

Enum `ProductType`: `SUBSCRIPTION` (закрытая группа, продлеваемая подписка) |
`LIFETIME` (общая группа, разовый платёж, бессрочно).

## 7. Интеграция с Робокассой (с фискализацией)

Робокасса сама формирует и отправляет чеки пользователям (облачная фискализация / Робочеки),
но чек `Receipt` обязан передаваться в платёжной ссылке и участвовать в подписи.

- **Платёжная ссылка:** `https://auth.robokassa.ru/Merchant/Index.aspx` с параметрами
  `MerchantLogin, OutSum, InvId, Description, Receipt, SignatureValue, IsTest`.
- **Receipt** — JSON вида:
  ```json
  {
  	"sno": "usn_income",
  	"items": [
  		{
  			"name": "Подписка на 30 дней",
  			"quantity": 1,
  			"sum": 990.0,
  			"payment_method": "full_payment",
  			"payment_object": "service",
  			"tax": "none"
  		}
  	]
  }
  ```
  `sno` и `tax` уточнить под систему налогообложения владельца.
- **SignatureValue (запрос):** `md5(MerchantLogin:OutSum:InvId:Receipt:Password1)`,
  где Receipt — **URL-encoded один раз**; в самой ссылке Receipt кодируется **дважды**
  (известная особенность Робокассы — проверить в тестовом режиме).
- **Result URL (webhook):** POST `/robokassa/result`. Проверка: `md5(OutSum:InvId:Password2)`
  (Receipt в подписи Result не участвует; сравнение без учёта регистра).
  Ответ строго `OK{InvId}` — иначе Робокасса ретраит.
- **Success/Fail URL:** редирект в бота (`https://t.me/<bot>?start=paid` / `?start=fail`) — только UX.
- **Источник истины — только Result URL.** Идемпотентность: платёж уже `PAID` → ответить `OK` без повторного продления.
- `InvId` = `Payment.id`. Тестовый режим `IsTest=1` на этапе разработки.
- `Description` и `Receipt.items[].name` зависят от продукта: «Подписка на 30 дней»
  (SUBSCRIPTION) / «Разовый доступ в общую группу» (LIFETIME). Механика подписей и
  кодирования Receipt одинакова для обоих продуктов.
- Обработка webhook ветвится по `Payment.product` внутри той же транзакции:
  SUBSCRIPTION → продление `expiresAt` + unmute; LIFETIME → upsert `CommonAccess`,
  `expiresAt`/статусы/флаги напоминаний не трогать.

## 8. Нефункциональные требования

- «Оплата → продление → unmute» — одна транзакция Prisma.
- Рассылки ≤ 25 msg/sec, ошибка 403 (бот заблокирован) не роняет джобу.
- Логи платежей и mute/unmute обязательны (pino).
- `.env`: `BOT_TOKEN, GROUP_ID, COMMON_GROUP_ID, ROBO_LOGIN, ROBO_PASS1, ROBO_PASS2, DATABASE_URL, ADMIN_ID, ADMIN_PANEL_URL, INTERNAL_API_TOKEN, AUTH_SECRET`.
- Публичный HTTPS для Result URL → VPS + Nginx; бот и панель — отдельные процессы PM2.
- Таймзона cron — `Europe/Moscow` (или из настроек).

## 9. Вне скоупа (v1)

Возвраты платежей, произвольное число тарифов/групп (в v1 ровно два продукта:
подписка на закрытую группу и разовый доступ в общую), рекуррентные списания,
экспорт в Excel, email-уведомления.

## 10. Критерии приёмки

1. `/start` → кнопка → сумма → оплата (тест) → одноразовая ссылка приходит; чек формируется Робокассой.
2. Повторный webhook не продлевает подписку дважды.
3. Cron мьютит истёкших; они остаются в группе и не пишут; после оплаты mute снимается автоматически.
4. Напоминание активным — 1 раз в окне 3 дней; замьюченным — каждые 10 дней до оплаты.
5. В панели: вкладка «Ограниченные» показывает всех MUTED; mute/unmute из панели применяется в группе ≤ 1 мин.
6. Дашборд показывает выручку, статусы, конверсию.
7. Не-админ не имеет доступа ни к `/admin` в боте, ни к панели.
8. Оплата общей группы (тест) → одноразовая ссылка в общую группу; `expiresAt` не меняется;
   cron никогда не мьютит пользователя за общую группу; повторный webhook не создаёт
   второй `CommonAccess`.

## 11. Открытые вопросы

- Система налогообложения владельца (`sno`) и ставка НДС (`tax`) для чека.
- Кикать ли пользователей, замьюченных дольше N дней (например, 60)?
- Точная цена разового доступа в общую группу (`price_common`; в сиде — заглушка 500).
- Нужны ли админам отдельные уведомления/сводки по общей группе.
