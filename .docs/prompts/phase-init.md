# Промпт: Создание новой фазы

Скопируй текст ниже и вставь в Cursor / Claude Code.

---

Нам нужно спланировать новую фазу разработки.

Прочитай эти файлы:

@CLAUDE.md — архитектура и правила проекта
@.docs/prd.md — требования и модель данных
@.docs/phases/phases.md — роадмап всех фаз (Что/Почему/DoD каждой фазы)
@.docs/phases/\_status.md — статусы фаз: какие сделаны, какая следующая
@.docs/phases/ — прочитай существующие phase-N.md, чтобы понять что уже реализовано

Затем:

1. По \_status.md определи следующую фазу (первая ⏳ Ожидает, все зависимости которой ✅)
2. Возьми её описание из phases.md — это основа скоупа, не выдумывай сверх него
3. Раздели фазу на таски (каждый таск = 1–2 дня, 3–7 файлов)
4. Для каждого таска укажи: цель, файлы/компоненты, DoD

Перед тем как предложить план — проверь документацию на проблемы:

- Противоречия между prd.md, CLAUDE.md и phases.md (поля, статусы, правила)
- Неполные спецификации (алгоритм описан, но edge cases без обработки)
- Несуществующие API / синтаксис в код-примерах
- Расхождения Prisma-схемы (prd.md п.6) и того, что требуют таски
- Робокасса: формулы подписей и кодирование Receipt должны совпадать с CLAUDE.md п.3 дословно
- Auth.js v5: сессию создаёт только `authorize()` credentials-провайдера через `signIn()`. Если описан кастомный route, который сам «создаёт сессию» — это не сработает
- Всё, что сломает реализацию, если разработчик скопирует пример как есть

Если найдены проблемы — укажи их списком ПЕРЕД планом фазы:

⚠️ Найденные проблемы в документации:

[файл, строка] — описание — рекомендуемый фикс

Жди решения по каждой проблеме прежде чем продолжать.
Если проблем нет — напиши «Документация проверена, противоречий не найдено» и переходи к плану.

Жди моего подтверждения плана.
После подтверждения:

1. Создай файл `.docs/phases/phase-N.md` по структуре @.docs/phases/phase-example.md
2. Обнови строку фазы в `.docs/phases/_status.md` (статус, ссылка на файл)

Учитывай специфику проекта:

- Монорепо: `apps/bot` (grammY + Fastify), `apps/admin` (Next.js 15), `packages/db` (Prisma, MySQL)
- Для каждого таска указывай, какое приложение он затрагивает: bot / admin / db
- Панель: Server Actions (+ Zod + `auth()` в каждом), НЕ отдельные REST API Routes
- Панель: `'use client'` только для интерактивности; Client Components не ходят в БД
- Бот: блокировка = restrictChatMember (mute), никогда banChatMember
- Изменение Prisma-схемы = миграция в `packages/db`
- `telegram_id` — BigInt везде
- Настройки читаются из таблицы `Setting`, не кэшируются
- Панель → Telegram только через internal API бота (Bearer `INTERNAL_API_TOKEN`)
- Тесты на подписи Робокассы пишутся в фазе 3, не откладываются

Не пиши код. Только планирование.
Жди подтверждения перед созданием файла.
