# Промпт: Создание таска

Скопируй текст ниже и вставь в Claude Code.

---

Нужно создать TASK.md для следующего таска.

Прочитай эти файлы:

- @CLAUDE.md — архитектура и правила проекта
- @.docs/phases/\_status.md — статусы фаз, чтобы определить текущую
- @.docs/prompts/dod-global.md — глобальные критерии готовности

Затем:

1. По \_status.md определи активную фазу:
   - статус 🔄 В работе — если такая есть, бери её
   - если нет — первая ⏳ Ожидает, для которой уже создан `.docs/phases/phase-N.md`
2. Прочитай файл этой фазы и найди первый таск со статусом ⏳ Ожидает
3. Прочитай разделы @.docs/prd.md, которые прямо относятся к таску (платежи — п.7, cron — п.3.3–3.4, панель — п.5 и т.д.)
4. **Перед планированием проверь документацию по этому таску:**
   - Противоречия prd.md ↔ CLAUDE.md ↔ phase-N.md (поля, типы, правила)
   - Код-примеры, которые не работают как написаны (несуществующий API, неверный синтаксис)
   - Неполные спецификации, которые заблокируют реализацию
   - Робокасса: формулы подписей/кодирование Receipt сверить с CLAUDE.md п.3 дословно
   - Auth.js v5: сессию создаёт только `authorize()` провайдера через `signIn()`; кастомный route, «создающий сессию» сам — не сработает

   Если найдены проблемы — **укажи их списком ПЕРЕД планом таска**:

   ⚠️ Найденные проблемы:

   [файл] — описание — рекомендуемый фикс

   Жди решения по каждой проблеме прежде чем продолжать.
   Если проблем нет — напиши «Документация проверена» и переходи к плану.

5. Составь список конкретных файлов/компонентов для создания или изменения (с указанием workspace: bot / admin / db)
6. Определи, что явно не входит в этот таск (out of scope)
7. Определи точки риска реализации — конкретно для этого таска, не общими словами:
   - Где правильный код по виду похож на неправильный: подпись ссылки Робокассы содержит Receipt и Pass1, подпись webhook — НЕ содержит Receipt и берёт Pass2; их легко перепутать
   - Где нужна идемпотентность/атомарность, а наивная реализация «прочитал → проверил → записал» выглядит рабочей
   - Где спека требует несколько вариантов одновременно (два типа напоминаний, mute И unmute), а легко реализовать один
   - Где нужна graceful-деградация (403 от Telegram, юзер вышел из группы), а не исключение
   - Где BigInt сломает JSON-сериализацию

   Если рисков не нашёл — напиши «Точек риска не нашёл» и переходи дальше.

8. Напиши проверяемые критерии DoD
9. Жди моего подтверждения
10. После подтверждения создай/перезапиши `TASK.md` в корне проекта:

```markdown
# TASK: <название таска>

## Фаза

Phase N — <название фазы>

## Статус

🔄 В работе

## Цель

<что должно работать после этого таска>

## Что нужно создать/изменить

- `apps/bot/src/...` / `apps/admin/app/...` / `packages/db/...` — что делает
- ...

## Out of scope

- <что явно не входит>

## ⚠️ Точки риска при реализации

- <конкретная точка> — <почему легко сделать неправильно>

## Definition of Done

- [ ] <проверяемый критерий 1>
- [ ] <проверяемый критерий 2>
- [ ] TypeScript компилируется без ошибок
- [ ] Проверить @.docs/prompts/dod-global.md (релевантные разделы)
```

Учитывай специфику проекта:

- Указывай workspace каждого файла (bot / admin / db)
- Панель: Server Actions + Zod + `auth()` первой строкой каждого action; НЕ отдельные REST API Routes
- Панель: указывай тип компонента (Server/Client); Client Components не ходят в БД
- Бот: mute через restrictChatMember, никогда banChatMember; рассылки только через `services/notify.ts`
- Если меняется БД — Prisma migration в `packages/db`
- Если нужна форма — Zod-схема
- `telegram_id` — BigInt; настройки — из таблицы `Setting` при каждом использовании
- Панель → Telegram только через internal API бота (Bearer `INTERNAL_API_TOKEN`)

Не пиши код. Только планирование.
Жди подтверждения перед записью файла.
