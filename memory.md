# Проверка оплаты Робокассы — как будем тестировать

Коротко:

**1. Пароли** — да, всё верно: сейчас (пока разрабатываем и тестируем) вписываешь в `.env` **тестовую** пару из блока «Параметры проведения тестовых платежей» (не боевую из «Технических настроек»).

**2. Result URL** — вот тут не всё так просто, «сделать такой же, но localhost» не получится:

Робокасса стучится на Result URL **со своих серверов через интернет**. `localhost` — это твой собственный компьютер, снаружи он никому не виден, Робокасса туда достучаться физически не может.

Чтобы это заработало, нужен **публичный туннель** до твоего localhost, например [ngrok](https://ngrok.com):

```
ngrok http 3000
```

Он выдаст тебе публичный адрес типа `https://abc123.ngrok-free.app` — вот его (плюс твой путь `/robokassa/result`) и надо будет подставить в Result URL: `https://abc123.ngrok-free.app/robokassa/result`.

⚠️ Но есть важный момент: **поле Result URL в кабинете — одно на все платежи**, и боевые (от текущего quescha-бота), и тестовые. Отдельного поля для тестового Result URL нет (в отличие от паролей). Значит, пока ты держишь там ngrok-адрес — **старый рабочий бот на quescha.com перестанет получать уведомления об оплате** (пока ngrok жив и работает).

Поэтому практичнее сделать так:

1. Сейчас, на этой фазе, у нас в проекте webhook-код (`payments/webhook.ts`) ещё даже не написан (это следующие фазы) — тестировать через реальный кабинет Робокассы пока рано.
2. Когда дойдём до фазы вебхука — тестировать его сначала **вручную через curl/Postman**, руками эмулируя запрос Робокассы с правильной тестовой подписью на `localhost` — вообще не трогая кабинет Робокассы.
3. Только когда весь бот полностью готов и задеплоен на постоянном публичном домене — один раз меняем Result URL в кабинете на настоящий продовый адрес нового бота (не ngrok, не localhost) и делаем финальную проверку реальным тестовым платежом.

Так безопаснее: не сломаешь текущий работающий приём денег, пока новый бот ещё не готов.

---

ROBO_IS_TEST
Что это: флаг тестового режима Робокассы — попадает в ссылку как IsTest=1. Робокасса в этом режиме не списывает реальные деньги и разрешает оплату тестовыми картами.
Значение: true — верно, оставляй так на всё время разработки. В проде (после деплоя) переключится на false.
Важно (см. memory.md): этот флаг работает в паре с ROBO_LOGIN/ROBO_PASS1/ROBO_PASS2 — они должны быть тестовой парой паролей из блока «Параметры проведения тестовых платежей» в кабинете Робокассы, а не боевыми из «Технических настроек» (боевые сейчас использует старый бот на quescha.com — их трогать нельзя).

ROBO_SNO
Что это: система налогообложения владельца — идёт в чек (Receipt), Робокасса требует один из строгих enum-кодов: osn, usn_income, usn_income_outcome, envd, esn, patent.
Значение: usn_income — это должно быть реальной системой налогообложения владельца бизнеса (УСН «доходы»), не условность для теста. Судя по phase-3.md, это значение уже было подтверждено пользователем до старта фазы — если это так и есть, оставляй как есть.

ROBO_TAX
Что это: ставка НДС по позиции чека, тоже строгий enum: none, vat0, vat10, vat110, vat20, vat120.

Значение: none — корректно для УСН (нет НДС). Тоже должно соответствовать реальному налоговому режиму, а не быть тестовой заглушкой.

Что это НЕ решает
Эти три переменные влияют только на платёжную ссылку (что строит Task 3.1 — buildPaymentUrl). Они никак не связаны с Result URL / webhook — про это memory.md предупреждает отдельно: ROBO_IS_TEST=true не заменяет туннель. Даже с тестовым режимом, чтобы Робокасса реально достучалась до POST /robokassa/result на твоём localhost, нужен публичный туннель (ngrok), а его пока разворачивать рано — сначала (Task 3.3) протестируем webhook вручную через curl с поддельной, но правильно посчитанной тестовой подписью, вообще не трогая кабинет Робокассы.

Так что на данном этапе (Task 3.1) — эти три строки в .env можно смело оставить как есть, они уже корректны для локальной разработки.

Что сейчас запущено
Бот — локально, на localhost:3000 (grammY + Fastify), подключён к реальной MySQL.
Публичный туннель (localtunnel) — пробрасывает интернет к этому localhost:3000: https://tricky-vans-smash.loca.lt
Я уже проверил curl'ом, что POST на https://tricky-vans-smash.loca.lt/robokassa/result реально долетает до бота (пришёл 400 Bad Request — это правильный ответ на неверную подпись, значит маршрут живой).

Шаг 1 — прописать URL в кабинете Робокассы
Зайти в личный кабинет Робокассы → раздел настроек магазина (обычно «Настройки» → «Технические настройки»), и указать:

Result URL (URL на сервер): https://tricky-vans-smash.loca.lt/robokassa/result
Success URL и Fail URL — на них Робокасса редиректит браузер пользователя после оплаты; в проекте это deep-link на бота (t.me/<имя_бота>?start=paid / ?start=fail), их можно тоже указать там, если ещё не стоят.
Важно: убедиться, что в кабинете включён метод передачи POST для Result URL (а не GET) — код бота читает тело через @fastify/formbody, рассчитан на POST.

Шаг 2 — прогнать тестовую оплату
В .env должно быть ROBO_IS_TEST=true — тогда Робокасса обрабатывает платёж в тестовом режиме (без реального списания денег, но webhook всё равно летит по-настоящему).
В Telegram открыть бота → /start → «Оформить подписку» → нажать «Оплатить».
Бот должен создать Payment(PENDING) в БД и дать реальную ссылку на auth.robokassa.ru.
Пройти тестовую оплату на странице Робокассы (тестовые реквизиты карты — смотри в документации Робокассы для IsTest=1, либо в кабинете есть подсказка).
Робокасса сама постучится на Result URL → через туннель → в бота.

Шаг 3 — что проверять (DoD фазы 3)
В логах бота (в терминале, где npm run dev -w apps/bot) должна появиться строка "robokassa webhook: payment processed".
В БД (npx prisma studio -w packages/db): Payment.status стал PAID, paidAt проставлен; у User — expiresAt продлён, reminderSentAt/lastMutedRemindAt сброшены в null.
Ответ Робокассе должен быть строго OK{InvId} (это не увидишь глазами, но именно так должно быть — уже покрыто автотестом).
Нажать «Оплатить» и оплатить второй раз тем же способом (или Робокасса сама иногда дублирует webhook) — подписка не должна продлиться ещё раз.
Попробовать отправить на Result URL что-то с заведомо неверной подписью (например, тем же curl, который я уже гонял) — должен быть 400, Payment не должен создаваться/меняться.
Важные нюансы
URL туннеля временный — если я остановлю процесс localtunnel или он сам отвалится, адрес tricky-vans-smash.loca.lt перестанет работать и станет недействительным, при новом запуске будет уже другой адрес. Это нормально для разовой ручной проверки, но не для постоянного использования — держите это в уме, если проверка растянется на день/два.
cloudflared (Cloudflare Tunnel) не завершился — на этой сети он блокируется на уровне файрвола/антивируса, поэтому используем localtunnel.
